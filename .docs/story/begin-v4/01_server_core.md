# 第一章：Web 服务内核 (Hello World)

> 现代互联网的基石是 Web 服务器。作为架构师，你的第一步是构建能够监听网络端口、响应客户端请求的核心服务，并确保它具备长期运行的能力。

## 本章目标

*   理解服务端程序的入口与生命周期概念。
*   熟悉启动日志与端口监听的基本流程。
*   建立"守护进程"运行方式的认知。

在开始编写代码之前，让我们先理解 Web 服务器的本质。每当你在浏览器中打开一个网页，你的浏览器（客户端）就会向某台机器（服务器）发出一条请求。服务器持续监听某个**端口号**——就像一栋大楼里的门牌号码——等待客户端叩门。一旦有请求到来，服务器就处理并回复。这个过程周而复始，永不停息，这就是"服务器主循环"的由来。

Web 服务器之所以要"持续运行"，是因为它本质上是一种**守护进程（Daemon）**。与普通命令行程序"执行完就退出"不同，守护进程启动后会驻留在内存中，静静等待外界的信号。想象一个 24 小时营业的便利店收银台——哪怕没有顾客，收银员也需要在岗，随时准备服务。

## 1. 启动监听 (Hello Server)

我们不使用现成的框架，而是从 `main` 函数开始，模拟一个 Web 服务器的启动过程。

在真实的网络编程中，服务器启动时会经历以下阶段：创建 Socket → 绑定端口（`bind`）→ 开始监听（`listen`）→ 接受连接（`accept`）→ 处理请求 → 返回响应。这个示例用日志输出来模拟这些阶段，帮助你建立直觉。

<!-- check:build_only -->
```cangjie
// file: web_core.cj

main() {
    println(">>> [CangjieWeb] 框架内核正在初始化...")

    // 模拟绑定端口
    let port = 8080
    println(">>> 正在监听端口: ${port}")
    println(">>> 服务启动成功！等待连接...")

    // 模拟服务器主循环
    while (true) {
        // 在真实场景中，这里会 accept() 一个 socket 连接
        // 为了演示，我们暂停一下模拟空转
        sleep(Duration.second * 5)
        println("... [Heartbeat] 服务运行中 ...")
    }
}
```

**代码解析：**

- `let port = 8080`：端口号 8080 是开发环境中常用的替代端口（正式的 HTTP 端口是 80），将它存储在变量中方便后续复用和修改。
- `while (true)`：这是服务器"永不退出"的关键——一个无限循环使得程序不会在执行完一次后结束，而是不断重复监听的过程。
- `sleep(Duration.second * 5)`：用 `sleep` 模拟服务器空闲等待，在真实实现中这里会是阻塞式的 `accept()` 调用，等待客户端连接到来。
- `println("... [Heartbeat] ...")`：心跳日志是生产环境中判断服务是否存活的重要信号，监控系统通常会检测这类周期性输出。

**💡 核心要点**

- **端口（Port）**：操作系统用端口号区分同一台机器上不同服务的通信通道，范围是 0–65535，1024 以下为系统保留端口。
- **`while (true)` 主循环**：Web 服务器的标志性结构，保证进程持续存活，是"守护进程"的代码体现。
- **心跳机制**：定期输出存活信号，让运维监控系统能及时发现服务异常。
- **`sleep()` vs `accept()`**：本例用 `sleep` 模拟等待；真实场景中 `accept()` 会阻塞当前线程直到有新连接，效率更高。

## 2. 编译与运行

```bash
cjc web_core.cj -o server
./server
```

虽然它现在还不能处理真正的 HTTP 请求，但它已经具备了作为一个守护进程（Daemon）的基本形态。这是所有高性能服务器的起点。

使用 `cjc` 将源文件编译成可执行文件 `server`，然后直接运行。你会看到启动日志，之后每隔 5 秒出现一次心跳输出。按 `Ctrl+C` 可以终止进程——在生产环境中，这个信号会被优雅关机逻辑捕获，以确保正在处理的请求能够完成。

## 工程化提示

*   真实服务需依赖标准网络库或框架，示例仅演示启动结构。
*   主循环应提供退出与健康检查机制，避免无控制的死循环。
*   日志中建议携带服务名称、端口与版本信息。
