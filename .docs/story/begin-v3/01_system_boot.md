## 本章目标

*   了解系统启动程序的最小结构。
*   熟悉日志输出与启动流程的基本习惯。
*   建立"代码编译-部署-运行"的工程思维。

## 1. 启动信号

任何系统都需要一个入口。在仓颉中，`main()` 函数就是这个开关。

**什么是 `main()` 函数？**

程序是由一系列指令组成的，但计算机需要知道"从哪里开始执行"。`main()` 函数就是这个约定好的起点——当你运行一个仓颉程序时，运行时环境会自动找到并调用 `main()`，然后按顺序执行其中的代码。

就像一场音乐会有一个开场音一样，`main()` 就是整个程序的序曲。没有它，程序无从开始；有了它，后续所有的逻辑才能有序展开。

**`println` 的作用**

`println` 是"print line"的缩写，意思是"打印一行文字并换行"。它是我们与程序沟通的最基本方式：程序把想告诉我们的信息打印到屏幕（终端）上。在真实的嵌入式或服务端系统中，这些输出通常会被重定向到日志文件，以便后续排查问题。

<!-- check:run -->
```cangjie
// file: smart_home_boot.cj
main() {
    println(">>> 智能家居控制中枢正在启动...")
    println(">>> 系统自检完成。准备就绪。")
}
```

<!-- expected_output:
>>> 智能家居控制中枢正在启动...
>>> 系统自检完成。准备就绪。
-->

**代码解析**：程序从 `main()` 的第一行开始执行，依次调用两次 `println`。每次调用都向终端输出一行文字，并自动在末尾添加换行符。`>>>` 前缀是一种常见的日志习惯，用来区分"系统提示"与普通内容，让运维人员一眼就能识别启动阶段的关键信息。

*   **现实意义**：`println` 是最基础的日志输出工具。在实际开发中，清晰的日志是系统运维的第一道防线。

**💡 核心要点**

*   `main()` 是仓颉程序的唯一入口，每个可执行程序必须有且只有一个 `main()` 函数。
*   `println(...)` 用于向控制台输出一行文本，括号内可以是字符串字面量，也可以是变量或表达式。
*   仓颉语句不需要以分号 `;` 结尾（与 Java/C++ 不同），代码更加简洁易读。
*   程序按照代码从上到下的顺序依次执行，这称为"顺序执行"，是最基本的控制流。

## 2. 编译部署

将你的源码编译为可执行的固件。

**编译的本质**

我们写的 `.cj` 源文件是人类可读的文本，而计算机只能执行二进制指令。编译器（`cjc`）的工作就是把人类写的代码"翻译"成机器能直接运行的可执行文件。这个过程同时也会进行语法检查和类型检查，帮助我们提前发现错误。

就像建筑图纸需要工人按图施工才能变成真正的房子，源代码需要经过编译才能成为可运行的程序。

```bash
cjc smart_home_boot.cj -o boot_loader
./boot_loader
```

**命令解析**：
- `cjc smart_home_boot.cj`：调用仓颉编译器，指定要编译的源文件。
- `-o boot_loader`：指定输出的可执行文件名为 `boot_loader`（`-o` 是 output 的缩写）。
- `./boot_loader`：在当前目录（`./`）下运行编译好的程序。

当终端打印出启动信息时，意味着你的代码已经成功接管了硬件的控制权。

**💡 核心要点**

*   源代码（`.cj` 文件）→ 编译器（`cjc`）→ 可执行文件（二进制）→ 运行，这是标准的编译型语言工作流程。
*   编译时发现的错误称为"编译时错误"，比运行时才暴露的错误更容易定位和修复。
*   良好的日志输出习惯应在项目最开始就建立，后期补加往往代价高昂。

## 工程化提示

*   启动日志建议包含模块名与版本号，便于部署排查。
*   `main` 中只做启动编排，避免把业务逻辑塞入入口。
*   真实固件部署还需要配置管理与失败回滚机制。
