# 第二章：传感器数据 (变量与类型)

> 系统启动后，首要任务是感知环境。我们需要从各类传感器读取数据（温度、湿度、状态），并将其存储在内存中，为后续决策提供依据。

## 本章目标

*   理解可变与不可变变量在传感器数据中的用途。
*   掌握基础类型与类型推断的实践方式。
*   认识类型安全在嵌入式系统中的意义。

## 1. 定义环境参数 (变量)

现实世界的数据千变万化。

在编程中，**变量**就是内存里的一个"命名储物格"——你可以给它起一个有意义的名字，然后在里面放数据，并在需要时读取或更新。仓颉提供了两种变量声明方式，对应两种不同的现实需求：

*   **`let`（不可变变量）**：一旦赋值就不能更改，适合存放那些确定不会变化的数据，比如设备序列号、安装地点。这类似于"刻在铭牌上的信息"，出厂后就固定了。
*   **`var`（可变变量）**：可以在运行期间随时更新，适合存放动态变化的数据，比如实时温度、电量百分比。这类似于"显示屏上的数字"，会随时刷新。

**类型推断**：仓颉是强类型语言，但你不必每次都显式写出类型。如果编译器能从初始值推断出类型（例如 `"Living Room"` 显然是字符串），就可以省略类型标注。这一特性称为**类型推断**，让代码既安全又简洁。

*   有些是固定的（如设备ID），用 `let`。
*   有些是波动的（如实时温度），用 `var`。

<!-- check:run -->
```cangjie
main() {
    // 设备元数据
    let deviceId: String = "SENSOR-001"
    let location = "Living Room" // 自动推断为 String

    // 实时监控数据
    var currentTemp: Float64 = 23.5 // 温度需要小数精度
    var humidity: Int64 = 45        // 湿度通常为整数百分比
    var isOnline: Bool = true       // 设备在线状态

    println("=== 设备状态监控 ===")
    println("ID: ${deviceId} | 位置: ${location}")
    println("状态: 在线(${isOnline}) | 温度: ${currentTemp}°C | 湿度: ${humidity}%")

    // 模拟数据变化
    currentTemp = 24.2
    println("更新温度: ${currentTemp}°C")
}
```

<!-- expected_output:
=== 设备状态监控 ===
ID: SENSOR-001 | 位置: Living Room
状态: 在线(true) | 温度: 23.500000°C | 湿度: 45%
更新温度: 24.200000°C
-->

**代码解析**：
- `let deviceId: String = "SENSOR-001"`：显式声明类型为 `String`，并赋予不可变值。
- `let location = "Living Room"`：省略类型标注，编译器自动推断为 `String`。
- `var currentTemp: Float64 = 23.5`：声明可变浮点数，`Float64` 表示 64 位双精度浮点，适合需要小数点精度的温度值。
- `"状态: 在线(${isOnline})"`：字符串插值语法 `${}` 可以将变量的值直接嵌入字符串中，避免繁琐的字符串拼接。
- `currentTemp = 24.2`：更新可变变量的值（注意不需要再写 `var`，直接赋值即可）。

**💡 核心要点**

*   `let` 声明不可变变量（类似 Kotlin 的 `val`），赋值后不能修改；`var` 声明可变变量。
*   仓颉的主要基础类型：`String`（文本）、`Int64`（整数）、`Float64`（浮点数）、`Bool`（布尔值）。
*   字符串插值 `"...${变量名}..."` 是格式化输出的推荐方式，比手动拼接更清晰、更不易出错。
*   `Float64` 打印时默认会显示 6 位小数（例如 `23.5` 显示为 `23.500000`），这是 IEEE 754 浮点标准的标准格式化行为。

## 2. 强类型的意义

为什么仓颉要区分 `Float64` 和 `Int64`？

**类型系统是代码的第一道护栏。** 在嵌入式开发中，资源是宝贵的。类型系统帮助我们在编译阶段就发现"将电压（数字）赋值给日志（文本）"这类逻辑错误，避免系统在运行时崩溃。

类比现实：想象你有一个只能存放液体的瓶子。如果你试图把固体放进去，在你拧盖子的时候（编译阶段）就会发现放不进去，而不是等到使用时（运行阶段）才发现漏出来。仓颉的类型系统正是这样一个"形状匹配检查"机制。

不同类型代表不同的内存布局和操作语义：`Float64` 占用 8 字节并支持小数运算，而将一个字符串赋给它毫无意义。编译器拒绝这种操作，帮你在第一时间阻止逻辑错误传播到生产环境。

<!-- check:compile_error -->
```cangjie
main() {
    var currentTemp: Float64 = 23.5
    currentTemp = "Error" // ❌ 编译器拦截：防止脏数据污染系统
}
```

**错误解析**：上面的代码会在编译时产生类型不匹配错误。`currentTemp` 已被声明为 `Float64`，而 `"Error"` 是 `String` 类型。仓颉编译器会明确报告"类型不匹配"，让你在部署前就修正这个错误，而不是等到传感器在凌晨三点上报了一个 `"Error"` 字符串时，整个控制系统才崩溃。

**💡 核心要点**

*   仓颉是**静态强类型**语言：每个变量的类型在编译时确定，且不同类型之间不能随意赋值。
*   编译时错误 > 运行时错误 > 逻辑错误：越早发现错误，修复成本越低。类型系统就是在帮你"提前发现"。
*   `let` 变量不仅是"不可变"，还向阅读代码的人传递了一个重要信息："这个值是设计上就不应该被改变的"，有助于代码理解和防止误操作。

## 工程化提示

*   传感器数据建议统一单位与采样周期，避免误读。
*   在线状态等布尔值应集中管理，便于故障诊断。
*   类型检查能提前阻断脏数据，建议保留严格编译选项。
