# 第九章：并行处理 (并发编程)

> 智能家居系统需要同时处理多件事：一边采集温湿度，一边监控安防摄像头，还要响应用户的语音指令。如果按顺序一件件做，系统会卡顿。我们需要并发 (`Concurrency`) 来解耦时间轴。

## 本章目标

*   掌握并发任务的启动与同步流程。
*   理解共享资源保护与原子操作的价值。
*   学会将独立任务拆分为可并行模块。

## 1. 多任务并行 (Spawn)

我们需要同时启动温度监控和安防监控。

**什么是并发（Concurrency）？**

传统的顺序程序一次只能做一件事：做完 A 再做 B。但真实世界中许多任务需要**同时进行**——就像厨师在等水烧开的同时切菜，而不是等水烧开后再切菜。**并发**就是把多个独立的任务交织执行，让程序更高效地利用 CPU 时间。

仓颉采用 **M:N 线程模型**：M 个轻量级的仓颉"线程"（协程）映射到 N 个操作系统线程上运行。`spawn { ... }` 语法创建一个新的并发任务（轻量级线程），花括号内的代码会在新线程中独立执行，不会阻塞当前（主）线程。

**`Future` 和 `.get()`**

`spawn { ... }` 返回一个 `Future<T>` 对象，代表"一个将来会完成的计算"。调用 `future.get()` 会**阻塞当前线程**，等待对应的任务完成并返回结果。这是最基本的线程同步机制：主线程可以先启动多个并发任务，然后在合适的时机等待它们全部完成。

类比现实：就像项目经理（主线程）同时给多个工程师（并发任务）分配了任务（`spawn`），然后在周会（`t1.get()`、`t2.get()`）上等待每个人汇报完成情况。工程师们并行工作，互不干扰，总用时取决于最慢的那个人，而不是所有人时间的总和。

```cangjie
import std.time.*
import std.sync.*

func monitorTemperature() {
    println("🌡️ 温度监控服务已启动...")
    sleep(Duration.millisecond * 500) // 模拟耗时任务
    println("🌡️ 温度数据采集完成")
}

func monitorSecurity() {
    println("📹 安防监控服务已启动...")
    sleep(Duration.millisecond * 800)
    println("📹 安防画面无异常")
}

main() {
    println(">>> 系统服务并行启动中...")

    // 启动独立线程，不阻塞主程序
    let t1 = spawn { monitorTemperature() }
    let t2 = spawn { monitorSecurity() }

    println(">>> 主线程: 等待子系统就绪...")

    // 等待任务完成
    t1.get()
    t2.get()

    println(">>> 所有子系统启动完毕")
}
```

## 2. 共享资源保护 (Atomic)

假设有多个传感器同时向一个"总能耗计数器"上报数据。如果没有保护，计数会出错。

**并发的陷阱：竞争条件（Race Condition）**

当多个线程同时读写同一个变量时，如果没有任何保护机制，就会出现"竞争条件"——执行结果取决于线程的调度顺序，产生不确定的错误结果。

举个例子：如果 `totalPowerUsage` 是普通 `var` 变量：
1. 线程 A 读取当前值（假设为 100）。
2. 线程 B 也读取当前值（同样为 100，因为 A 还没写回）。
3. 线程 A 计算 100 + 50 = 150，写回。
4. 线程 B 计算 100 + 50 = 150，写回。
5. 最终结果是 150，而正确结果应该是 200。

**原子操作（Atomic）的原理**

`AtomicInt64` 提供的操作（如 `fetchAdd`）在硬件层面保证了"读-改-写"是**原子的**（不可被中断的），即使多个线程同时调用，每次操作也都是完整的，不会相互干扰。这是解决简单计数器竞争条件的最高效方法。

类比现实：原子操作就像一个特别的收银台，一次只接受一个顾客付款（原子性），即使排队的人再多，钱款也不会算错。普通变量则像把钱放在桌上，任何人都可能同时去拿，造成混乱。

<!-- check:run -->
```cangjie
import std.sync.*
import std.collection.*

main() {
    // 全屋总能耗 (原子变量，线程安全)
    let totalPowerUsage = AtomicInt64(0)
    let tasks = ArrayList<Future<Int64>>()

    println("开始统计全屋能耗...")

    // 模拟 10 个设备同时上报能耗，每个消耗 50W
    for (_ in 0..10) {
        let f = spawn {
            sleep(Duration.millisecond * 10)
            totalPowerUsage.fetchAdd(50) // 原子加法
        }
        tasks.add(f)
    }

    // 等待统计完成
    for (f in tasks) { f.get() }

    println("当前实时总功率: ${totalPowerUsage.load()} W") // 结果应为 500
}
```

```bash
开始统计全屋能耗...
当前实时总功率: 500 W
```

**代码解析**：
- `let totalPowerUsage = AtomicInt64(0)`：创建一个初始值为 0 的原子整数变量。原子类型而非普通 `var` 是关键——确保多线程并发访问的正确性。
- `for (_ in 0..10)`：循环 10 次，`_` 表示不关心循环变量的值（占位符）。
- `spawn { sleep(...); totalPowerUsage.fetchAdd(50) }`：每次循环启动一个新线程，该线程等待 10 毫秒后原子地将总能耗加 50W。10 个设备 × 50W = 500W。
- `tasks.add(f)`：把每个 `Future` 存入列表，以便后续统一等待。
- `for (f in tasks) { f.get() }`：遍历所有 `Future`，确保所有设备上报完成后再读取总值。
- `totalPowerUsage.load()`：安全地读取原子变量的当前值（`.load()` 同样是原子操作）。

**💡 核心要点**

*   **`spawn { ... }`**：创建轻量级并发任务，返回 `Future<T>`。任务中的代码与调用方并行执行。
*   **`future.get()`**：等待并发任务完成，获取返回值（阻塞直到任务结束）。是控制任务完成顺序的基本手段。
*   **`AtomicInt64`**：提供线程安全的整数操作（`fetchAdd`、`load`、`store` 等），适用于并发场景下的简单计数器。对于复杂的共享状态，应使用互斥锁（`Mutex`）。
*   并发编程的核心挑战：多个线程共享数据时必须进行同步，否则会产生竞争条件（结果不确定）。选择合适的同步原语（原子操作、锁、消息队列等）是并发编程的关键技能。

## 工程化提示

*   并发任务应设置超时与失败回调，避免阻塞主流程。
*   共享资源计数可用原子变量，复杂状态请使用锁或消息队列。
*   并发输出日志可能乱序，建议采用统一日志队列。
