# 第三章：自动化逻辑 (流程控制)

> 智能家居的核心在于"自动决策"。如果温度过高，自动开空调；如果没人在家，自动关灯。我们需要编写这些控制逻辑，并确保策略具备可解释性与可调节性。

## 本章目标

*   掌握条件分支与循环控制的基础语法。
*   学会将传感器阈值转化为可执行策略。
*   形成"循环监控 + 及时退出"的设计习惯。

## 1. 阈值判断 (if-else)

根据传感器数据做出响应。

**什么是条件分支？**

程序的执行并不总是一条直线。现实世界的决策充满"如果……那么……否则……"的逻辑。`if-else` 语句正是这种决策逻辑的直接翻译。

就像家里的温控器：当温度高于设定值时自动打开空调，低于时打开暖气，否则保持待机。这种"根据条件选择不同路径执行"的能力，是赋予程序"智能"的核心机制之一。

**多分支判断**：当有多个互斥的条件需要判断时，可以用 `if ... else if ... else` 链来处理。编译器会从上到下依次检查条件，一旦找到第一个为 `true` 的条件，就执行对应的代码块，然后跳过其余所有分支。

<!-- check:run -->
```cangjie
main() {
    let temp = 31.0
    let userSetTemp = 26.0

    print("当前室温: ${temp}°C -> ")

    if (temp > userSetTemp + 2.0) {
        println("❄️ 降温模式：开启空调")
    } else if (temp < userSetTemp - 2.0) {
        println("☀️ 取暖模式：开启暖气")
    } else {
        println("🍃 节能模式：维持现状")
    }
}
```

<!-- expected_output:
当前室温: 31.000000°C -> ❄️ 降温模式：开启空调
-->

**代码解析**：
- `temp > userSetTemp + 2.0`：判断当前温度是否比设定值高出 2 度以上。加入 `2.0` 的"缓冲区间"是一种工程实践，避免温度在临界值附近频繁触发切换（称为"抖动"）。
- `print(...)` 与 `println(...)` 的区别：`print` 输出文字后**不**换行，`println` 则会在末尾自动换行。此处 `print("当前室温: ...")` 先输出前半部分，再由 `println` 输出后半部分，两者在同一行显示。
- `else` 分支：当所有 `if` 和 `else if` 条件都不满足时，才执行 `else` 块，这是保底的默认行为。

**💡 核心要点**

*   `if` 条件表达式的括号是必须的：`if (条件) { ... }`。
*   条件可以是任意返回 `Bool` 值的表达式，例如比较运算符（`>`、`<`、`==`、`!=`）或逻辑运算符（`&&`、`||`、`!`）的组合。
*   为阈值判断加入"迟滞区间"（如 `± 2.0`）是嵌入式控制系统的常见工程实践，可以有效防止设备频繁开关导致的损耗。

## 2. 持续监控 (while 循环)

传感器需要全天候工作。我们需要一个主循环来不断轮询数据。

**什么是循环？**

许多任务需要反复执行：每隔一秒读取一次传感器数据、不断检查是否有新的用户指令……如果手动写 1000 次相同的代码显然不现实，这时就需要**循环**。

`while` 循环的逻辑非常直接："只要条件为真，就一直重复执行循环体内的代码"。它就像一个不知疲倦的值班员，条件不满足（比如电量耗尽）才会下班。

**`break` 和 `continue` 的作用**：
- `break`：立即终止整个循环，跳出循环体，继续执行循环后面的代码。常用于"满足某个紧急条件时提前退出"。
- `continue`（在后续 for-in 示例中出现）：跳过本次循环剩余的代码，直接进入下一次循环迭代的条件判断。

<!-- check:run -->
```cangjie
main() {
    var checkCount = 0
    var batteryLevel = 100

    println(">>> 开始持续监控...")

    // 模拟运行 5 个周期
    while (checkCount < 5) {
        if (batteryLevel < 10) {
            println("⚠️ 电量低！系统进入休眠。")
            break
        }

        checkCount = checkCount + 1
        batteryLevel = batteryLevel - 15
        println("第 ${checkCount} 次轮询: 状态正常 (电量 ${batteryLevel}%)")
    }
}
```

<!-- expected_output:
>>> 开始持续监控...
第 1 次轮询: 状态正常 (电量 85%)
第 2 次轮询: 状态正常 (电量 70%)
第 3 次轮询: 状态正常 (电量 55%)
第 4 次轮询: 状态正常 (电量 40%)
第 5 次轮询: 状态正常 (电量 25%)
-->

**代码解析**：
- `while (checkCount < 5)`：循环条件是"已轮询次数小于 5"，每次循环结束后 `checkCount` 加 1，直到等于 5 时条件为假，循环自然结束。
- `if (batteryLevel < 10) { break }`：在每次轮询开始前先检查电量，如果过低则立即中断循环，模拟电量告警保护机制。在本例中由于初始电量 100 每次减少 15，5 次后还剩 25%，所以 `break` 始终不会触发。
- `checkCount = checkCount + 1`：手动递增计数器。仓颉也支持 `checkCount += 1` 的简写形式。

## 3. 设备自检 (for-in 循环)

系统启动时，需要遍历所有已连接的端口进行检查。

**`for-in` 循环：为遍历而生**

`while` 循环非常灵活，但需要手动管理计数器，稍不注意就会写出"死循环"或"差一错误"（off-by-one error）。当你需要遍历一个已知范围或集合中的每个元素时，`for-in` 是更安全、更易读的选择。

`1..=4` 是仓颉的**区间字面量**，表示从 1 到 4 的闭区间（包含两端）。如果写成 `1..4`，则表示半开区间（包含 1，不包含 4）。`for-in` 会依次将区间中的每个值赋给循环变量 `portId`，然后执行一次循环体。

类比现实：`for-in` 就像工厂流水线上的质检员，拿着一个清单（区间或集合），一个接一个地检查，检查完就打钩，直到清单上的所有项目都处理完毕。

<!-- check:run -->
```cangjie
main() {
    println("正在扫描端口 1-4...")

    for (portId in 1..=4) {
        // 假设端口 3 是调试接口，跳过
        if (portId == 3) {
            continue
        }
        println("✅ 端口 ${portId}: 连接正常")
    }
}
```

<!-- expected_output:
正在扫描端口 1-4...
✅ 端口 1: 连接正常
✅ 端口 2: 连接正常
✅ 端口 4: 连接正常
-->

**代码解析**：
- `for (portId in 1..=4)`：`portId` 依次取值 1、2、3、4。`..=` 表示闭区间，两端都包含。
- `if (portId == 3) { continue }`：当 `portId` 等于 3 时，执行 `continue` 跳过本次循环剩余部分，直接进入下一次迭代（`portId = 4`）。因此输出中没有"端口 3"的记录，模拟了"跳过调试接口"的逻辑。
- 输出顺序是 1、2、4，3 被跳过，完整体现了 `continue` 的效果。

通过这些逻辑，原本冰冷的硬件开始表现出"智能"。

**💡 核心要点**

*   `while (条件)` 适合"不知道要循环几次"的场景（如等待网络连接）；`for-in 区间/集合` 适合"已知遍历范围"的场景。
*   `break`：立即跳出当前循环；`continue`：跳过本次迭代，继续下一次。
*   区间字面量：`a..b` 为半开区间（不含 b），`a..=b` 为闭区间（含 b）。
*   循环中的 `if` 判断通常充当"守卫条件"，在满足特殊情况时改变循环的行为，这是控制流的常见组合模式。

## 工程化提示

*   阈值判断要考虑"抖动区间"，避免频繁开关设备。
*   监控循环应设置休眠间隔，避免占满 CPU。
*   设备自检结果建议写入日志或上报平台。
