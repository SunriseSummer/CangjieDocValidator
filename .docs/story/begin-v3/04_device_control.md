# 第四章：设备驱动 (函数与闭包)

> 随着接入设备增多，直接写 `if-else` 会让代码变得难以维护。我们需要将特定设备的控制逻辑封装成"驱动函数"，形成统一可复用的控制层。

## 本章目标

*   学会用函数封装设备控制指令。
*   理解高阶函数在场景编排中的价值。
*   建立"业务与硬件解耦"的设计意识。

## 1. 封装控制指令 (定义函数)

将"打开空调"这个动作封装起来，无论在哪个房间，调用方式都一样。

**为什么需要函数？**

想象你的控制系统有 10 个房间，每个房间都需要控制空调。如果每次都把完整的空调控制代码复制粘贴 10 遍，代码就会变得臃肿，而且一旦空调控制协议升级，你需要修改 10 个地方，极易遗漏。

**函数（Function）** 是解决这个问题的核心工具：将一段有具体功能的代码块命名，然后在需要的地方通过名字来"调用"它。这样，控制逻辑只需要写一次，在任何地方都可以复用。修改时也只需要改一处。

**函数的三要素**：
- **函数名**：有意义的名字，描述这个函数"做什么"（如 `setAirConditioner`）。
- **参数**：函数需要的输入信息（如控制模式 `mode` 和目标温度 `temp`）。
- **返回值**：函数执行后输出的结果（若无返回值，类型为 `Unit`，可以省略不写）。

<!-- check:run -->
```cangjie
// 定义驱动函数
func setAirConditioner(mode: String, temp: Int64) {
    println("📡 发送指令 -> 空调: 模式[${mode}], 设定温度[${temp}]")
    // 这里可以是真实的硬件 I/O 操作
}

main() {
    // 业务逻辑调用，无需关心底层硬件细节
    setAirConditioner("制冷", 24)
}
```

<!-- expected_output:
📡 发送指令 -> 空调: 模式[制冷], 设定温度[24]
-->

**代码解析**：
- `func setAirConditioner(mode: String, temp: Int64)`：定义一个函数，接受两个参数：`mode`（字符串类型，控制模式）和 `temp`（整数类型，目标温度）。
- 函数体内的 `println` 输出了一条控制日志，在实际系统中这里会替换为真正的硬件通信代码（I/O 操作、串口发送等）。
- `setAirConditioner("制冷", 24)`：调用函数时，按顺序传入实际的参数值（称为"实参"）。参数名（"形参"）和实参值通过位置对应。

**💡 核心要点**

*   函数实现了代码的"一次编写，多处调用"，是消除重复代码的最基本手段。
*   仓颉函数参数支持**具名参数**（调用时写出参数名），有助于提高调用代码的可读性。
*   函数内部的注释 `// 这里可以是真实的硬件 I/O 操作` 是一种良好习惯：用注释说明扩展点，方便后来的开发者理解意图。

## 2. 场景模式 (高阶函数)

用户常定义"离家模式"或"观影模式"。这些模式本质上是一系列指令的组合。我们可以编写一个"场景执行器"。

**什么是高阶函数？**

如果说普通函数接受"数据"作为参数，那么**高阶函数**接受"函数"作为参数（或返回一个函数）。这是一种强大的抽象机制，让你可以把"行为"本身当作数据传递。

类比现实：`executeScene` 就像一个活动策划公司——你告诉它"今晚要办什么活动"（场景名），并提供一个"活动方案"（具体执行的函数），策划公司负责宣布开始、执行方案、宣布结束。策划公司本身不关心具体活动内容，只负责流程编排。

**Lambda 表达式**：`{ => ... }` 是仓颉中定义匿名函数（Lambda）的语法。你可以把一段代码定义为"值"并赋给变量（如 `let awayMode = { => ... }`），然后像传递普通变量一样把它传给高阶函数。

`() -> Unit` 是函数类型的写法：`()` 表示无参数，`Unit` 表示无返回值。这告诉编译器"我接受一个没有参数且不返回任何值的函数"。

<!-- check:run -->
```cangjie
// 场景执行器：接收一个函数作为具体场景逻辑
func executeScene(sceneName: String, action: () -> Unit) {
    println(">>> 正在激活场景: ${sceneName}")
    action() // 执行传入的逻辑
    println(">>> 场景激活完毕\n")
}

main() {
    // 定义离家模式 (Lambda)
    let awayMode = { =>
        println(" - 关闭所有灯光")
        println(" - 开启安防摄像头")
        println(" - 锁定大门")
    }

    // 执行
    executeScene("离家模式", awayMode)
}
```

<!-- expected_output:
>>> 正在激活场景: 离家模式
 - 关闭所有灯光
 - 开启安防摄像头
 - 锁定大门
>>> 场景激活完毕
-->

**代码解析**：
- `func executeScene(sceneName: String, action: () -> Unit)`：第二个参数 `action` 的类型是 `() -> Unit`——一个没有参数、没有返回值的函数。这就是高阶函数的参数声明方式。
- `action()`：在执行器内部，直接用参数名后跟 `()` 即可调用传入的函数，就像调用普通函数一样。
- `let awayMode = { => ... }`：Lambda 语法创建了一个匿名函数并绑定到 `awayMode` 变量。`=>` 左边是参数列表（此处为空），右边是函数体。
- `executeScene("离家模式", awayMode)`：把 `awayMode` 这个函数作为参数传入，`executeScene` 会在合适的时机调用它。

通过函数，我们将"做什么"（业务）和"怎么做"（实现）分离开来，这是现代软件工程的基石。

**💡 核心要点**

*   **高阶函数**：接受函数作为参数，或返回一个函数的函数。它让你把"逻辑"本身作为数据传递，实现灵活的行为组合。
*   **Lambda 表达式**：`{ 参数 => 函数体 }` 语法定义匿名函数，无需单独命名，适合临时使用的简短逻辑。
*   **函数类型**：`(参数类型1, 参数类型2) -> 返回类型` 是仓颉中描述函数签名的类型写法。
*   函数式编程的核心思想：将变化的部分（具体场景逻辑）与稳定的部分（场景执行框架）分离，各自独立演化，降低代码耦合度。

## 工程化提示

*   设备驱动函数应保持幂等性，避免重复指令造成异常。
*   场景模式建议记录执行日志，便于回放与故障分析。
*   高阶函数的参数命名要直观，减少调用端误用。
